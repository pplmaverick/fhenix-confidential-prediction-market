# Draft GitHub comment — cofhesdk issue #272

Status: draft, not yet posted.
Target: https://github.com/FhenixProtocol/cofhesdk/issues/272

---

**Environment**
- `@cofhe/sdk`: `0.6.0`
- Client: `createCofheClient` via `@cofhe/sdk/web`
- Chain: Arbitrum Sepolia (421614)
- Permit type: self permit, created via `cofheClient.permits.getOrCreateSelfPermit()`

**Symptom**

After a permit has been idle for a few days (default expiration is 7 days), every subsequent `decryptForView(...).withPermit().execute()` call fails with:

```
Error: Permit is expired
```

This happens even though the app calls `getOrCreateSelfPermit()` on every page load before attempting any decryption. The permit is never refreshed — it fails the same way indefinitely, every session, until the user (or app) manually clears it.

A related symptom in our case: our app treats a client-connect step as "CoFHE ready" independently of permit creation (permit creation failures are caught and logged separately so they don't block betting), so this failure mode is easy to miss — the UI reports the CoFHE client as ready while every permit-gated decrypt silently fails behind it.

**Root cause**

`getOrCreateSelfPermit()` only checks whether an active self permit *exists* — it does not check whether that permit is still *valid*:

```js
// @cofhe/sdk 0.6.0, permit implementation
var getOrCreateSelfPermit = async (publicClient, walletClient, chainId, account, options) => {
  const _chainId = chainId ?? await publicClient.getChainId();
  const _account = account ?? walletClient.account.address;
  const activePermit = await getActivePermit(_chainId, _account);
  if (activePermit && activePermit.type === "self") {
    return activePermit;
  }
  return createSelf(options ?? { issuer: _account, name: "Autogenerated Self Permit" }, publicClient, walletClient);
};
```

The `if (activePermit && activePermit.type === "self")` check has no expiry condition, so once a permit is stored as "active" for an account/chain, `getOrCreateSelfPermit()` keeps returning that same object forever, even long after `permit.expiration` has passed. Expiry checking exists elsewhere in the SDK (`PermitUtils.isExpired`, `ValidationUtils.isExpired`), it's just not consulted here.

**Reproduction steps**

1. Connect a wallet and call `cofheClient.permits.getOrCreateSelfPermit()` to create and sign a self permit (default 7-day expiration).
2. Wait until `permit.expiration` has passed (or manually back-date it for testing).
3. Reload the app and call `cofheClient.permits.getOrCreateSelfPermit()` again.
4. Observe that it resolves successfully and returns the same (expired) permit object, with no error or warning.
5. Call `cofheClient.decryptForView(ctHash, utype).withPermit().execute()` using that permit.
6. Observe the call rejects with `Permit is expired`, and will continue to do so on every retry, since step 3 never replaces the stale permit.

**Workaround**

Before calling `getOrCreateSelfPermit()`, check the currently active permit's expiration and evict it if it has passed, so the SDK is forced to sign a fresh one:

```ts
const existingPermit = cofheClient.permits.getActivePermit(chainId, account)
if (existingPermit && existingPermit.expiration < Math.floor(Date.now() / 1000)) {
  await cofheClient.permits.removePermit(existingPermit.hash, chainId, account)
}
await cofheClient.permits.getOrCreateSelfPermit()
```

**Suggested fix**

`getOrCreateSelfPermit()` (and likely `getOrCreateSharingPermit()`, which has the same pattern) could check `ValidationUtils.isExpired(activePermit)` alongside the existing `type === "self"` check, and fall through to `createSelf(...)` when the active permit is expired — removing the need for callers to implement this themselves.

---

I've implemented the workaround above in production and confirmed it resolves the issue — decrypts that were failing with `Permit is expired` now succeed after the stale permit is evicted and a fresh one is signed.
